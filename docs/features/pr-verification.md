# Multi-Model PR Verification

One of ContriBot's most powerful features is its autonomous Pull Request verification system. Instead of relying on a single AI model, ContriBot uses a panel of two distinct models to review code.

## The 2-Model Verification System
When a PR is opened (by ContriBot or a human), the following models review the diff:
1. **Gemini 3.1 Pro**: Focuses on deep architectural flaws and complex logic errors.
2. **Gemini 3.0 Flash**: Checks for performance issues and best practices.

## How Consensus Scoring Works
Each model independently evaluates the PR and casts a vote: `Approved` (True/False) and a `Score` (1-10).
- **Consensus Score**: The number of models that approved the PR (0 to 2).
- **Safe to Merge**: A PR is marked as "Safe to Merge" only if it achieves a consensus score of **2**.

## What Each Model Checks
- **Bugs**: Logic errors, off-by-one errors, null pointer exceptions.
- **Security**: SQL injection, XSS, exposed secrets.
- **Best Practices**: DRY principles, SOLID principles, proper error handling.

## Understanding Verification Results
The verification results are posted as a comment on the GitHub PR and are visible in the ContriBot UI. The summary includes:
- The overall consensus score.
- A concise summary generated by a lightweight model (Gemini Flash Lite) combining the feedback of both models.
- Specific issues found and suggestions for improvement.

## The Merge Workflow
**Crucial Note**: ContriBot will **never** merge a Pull Request automatically. The final decision always rests with a human maintainer. ContriBot provides the verification data to help you make an informed decision quickly.
